# AI로 하는 기획

## 1. 인트로 & 오프닝

**AI로 하는 기획 — 생산성 향상과 프론트엔드 협업 사례 공유**

### 공유회 목적

1. 정답 제시 (X) → 먼저 겪은 **'시행착오' 공유**를 통한 러닝 커브 단축 (O)
2. 특정 툴(Kiro) 홍보 (X) → AI 에이전트는 상황에 맞는 **'선택지'**일 뿐

---

## 2. Why: 우리는 왜 기존 툴을 버렸나?

### 기존에 사용하던 도구들

- **Axure** — 인터랙션은 좋으나 결국 '가짜 로직(Pseudo-code)'. 구현을 위한 인적·시간 리소스 낭비.
- **Figma** — 철저한 '디자인' 도구. 상태(State)와 로직을 텍스트로 설명해야 함 → 동기화 실패의 연속.

### 핵심 Pain Point

> 기획 산출물은 결국 개발자가 **또 다시 코드로 번역**해야 한다!

- 디자이너·퍼블리셔 부재 → 프론트엔드 개발자가 독박 코딩
- 기획 ↔ 개발 사이의 번역 비용이 막대

### 그래서 원했던 것

1. **Pain Point 해소** — 기획-개발 간 번역 비용 제거
2. **퀄리티 향상** — AI 에이전트와 함께하는 기획으로 생산성 + 품질 동시 달성

### 해답: Docs as Code

> **Kiro + React**

- 기획서 = 멈춰있는 문서가 아닌 **'실제 동작하는 코드'**
- **단일 진실 공급원 (Single Source of Truth)** 달성

## 3. How: AI 기획의 오해와 진실 (🚨 실망주의)

### 한 줄 요약

> "실망스러우시겠지만, **'뚝딱' 만들어지는 마법은 없습니다.**"

### 빙산 모델 (Iceberg Model)

- **수면 위 (10%)** — AI가 생성한 깔끔한 UI 화면
- **수면 아래 (90%)** — 기획자가 설계한 **'하네스 엔지니어링(Harness Engineering)'**

---

## 4. 방법론: 하네스 엔지니어링 (Harness Engineering)

### 정의

> 🐴 **야생말 비유** — AI는 본능대로 날뛰는 야생말.
> 의도대로 정확히 달리게 하려면 촘촘한 **마구(Harness)**가 필요하다.

### 해결해야 할 '날뛰는 야생마' 2가지

1. **컨텍스트 부패 (Context Decay)** — 작업이 길어지면 AI가 앞선 정책을 잊어버리는 현상
2. **규칙과 울타리 부재** — 지능이 부족해서가 아니라, '하면 안 되는 것'에 대한 구조적 제약이 없을 때 생기는 참사

---

## 5. How: AX팀의 하네스 3기둥 구축 사례

### 제1기둥: 컨텍스트 문서 (절대 규칙 지도)

- 기술 스택 (React, TypeScript) 명시
- UI 컴포넌트 (Shadcn, MUI) 강제 매핑
- 폰트 및 컬러 정책 사전 주입
- 기획 문서 일체 포함 — IA, 정책서, 기능 정의서, UI 스켈레톤

### 제2기둥: 자동 강제 시스템 (Automated Enforcement)

- 에이전트가 룰을 어기면 **저장을 막고 다시 뱉어내게** 하는 자동 교정 루프
- Steering 제어를 통한 실시간 규칙 강제

### 제3기둥: 가비지 컬렉션 (Garbage Collection)

- AI가 만들어낸 나쁜 코드 패턴(가비지)을 **주기적으로 청소**
- 오학습 방지 및 코드베이스 품질 유지


---

## 6. How: 협업 방식

### 1) 기획자 간 협업

- Code 베이스 → 당연히 **Git** 활용
- 단, 브랜치는 **개별(개인별)** 운영
- 공통 레이아웃과 컴포넌트를 **사전 정의하여 제공**
- 협업 규칙을 **md 파일로 작성**하여 공유

### 2) 기획 → FE 개발자 핸드오프

- 기획서는 **Vercel로 배포** → URL 형태로 전달
- **레파지토리를 통째로 제공**
- 개발자는 **CSS 산출물을 그대로 사용** → 비즈니스 로직 최적화와 API 연동까지 가는 길을 혁신적으로 단축
- 기획자와의 핑퐁, 코딩 횟수 **혁신적으로 감소!**
